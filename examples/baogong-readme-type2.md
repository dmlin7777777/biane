## 一个问题

你写了一段经历："负责数据分析相关工作，得到领导认可"。

面试官看了一眼，问："具体做了什么？数据量有多大？影响了什么决策？"
你说不出——因为你写简历的时候没想这些。

**你需要的不是润色器。你需要的是一个会追问你的人。**

**这两个版本的差距，就是面试中"说不清"和"说得信"的差距。**

每一行数字，都有来源。每一条动词，都有证据。包公不编数字——量化追问最多 2 轮，说不出来就保留原文。

## 你什么时候用它

- 你拿到了一个 JD，想把简历往上面靠 —— 说一句"帮我针对这个 JD 调简历"
- 你投了多个岗位，每份简历要不同侧重 —— A2 批量定制 + 跨 JD 差异对比表
- 你没有 JD，只想打磨通用简历 —— Mode B：故事库驱动的能力挖掘
- 你还没简历，但有经历想写 —— 从头引导搭建，5 个问题建好 master 简历
- 你拿到一个 JD 想先看看自己匹配不匹配 —— Scenario C：纯 JD 分析，不写简历

**6 个场景，自适应路由。你不用选模式——说什么就走什么管线。**

## 铡刀门：编造阻断

**不是"建议"——是硬门控。违反一条，整份草稿打回。**

| 铁律 | 怎么检查 |
|---|---|
| **数据必须有来源** | "提升 30%" → 你怎么知道？来自故事库，还是用户亲口确认？ |
| **动词必须匹配证据** | "主导" → 你是最终决策者？还是一句简单的"参与"？ |
| **推断禁止交付** | `[~]` 模型推断 → 自动升级一个风险等级 → 禁止出现在终稿 |
| **撰写/审计物理隔离** | Writer 只写、Auditor 只审，两个独立 LLM 调用，**不能自己审自己** |

| 信息状态 | 标记 | 面试可信度 |
|---|---|---|
| 用户亲口确认 | `[✓]` | 最高——你答得出来 |
| 待本轮确认 | `[?]` | 需验证——上下文可能变了 |
| 模型推断 | `[~]` | **禁止交付**——你答不出来 |

## 包公不做的事

- **不编数字。** 2 轮追问后你给不出数字 → 保留原文，绝不编造
- **不编经历。** 检测到"帮我编一段实习" → 硬阻断 + 替代建议
- **不自我审计。** Writer 和 Auditor 是两个独立调用，不会出现"自己审自己"
- **不删你文件。** 源简历 `resume_master.md` 全程只读
- **不替你投递。** 交付后你自己审、自己投
- **不声称完美。** 如果你的背景和 JD 差距太大，包公会直接告诉你"这条不建议硬凑"
